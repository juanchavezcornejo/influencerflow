---
description: Generate a new Alembic revision and review the diff
argument-hint: "migration message, e.g. add users table"
---

Generate a new autogenerated Alembic revision.

1. Take the user's argument as the migration message. If empty, ask for one
   before proceeding — revisions without a message are painful later.
2. Run `make migrate-new msg="<message>"`.
3. Read the newly-created file under `api/alembic/versions/`.
4. Review it:
   - Does it match the ORM change the user just made?
   - Any `op.drop_*` calls that look unintentional? (Autogenerate can
     miss server defaults, enum changes, indexes.)
   - Are column types what you expect?
5. Report back with the file path + a short summary, and flag anything
   suspicious.

If the user wants to apply it immediately: `make migrate`.
