# MIGRATIONS.md — Alembic Migration Guide

Database schema management for InfluencerFlow using Alembic + SQLAlchemy async.

---

## Workflow

Follow these five steps for every schema change:

1. **Modify the SQLAlchemy model** in `api/app/models/`.
2. **Generate a migration file**: `make migrate-new msg="describe what changed"` — runs `alembic revision --autogenerate`.
3. **Review the generated file** in `api/alembic/versions/`. Check for anything Alembic gets wrong (see "What Needs Manual Review" below). Rename to match the naming convention if needed.
4. **Apply the migration**: `make migrate` — runs `alembic upgrade head` inside the running `api` container.
5. **Commit migration + model together** in the same commit. Never commit one without the other.

---

## Naming Convention

Files follow the pattern `NNN_description.py` with a zero-padded 3-digit sequence number:

```
001_initial_schema.py
002_groups_and_grouping.py
003_add_near_duplicate_clustering.py
```

When Alembic autogenerates a file, rename it to match this convention before committing. The autogenerated hash-prefixed filename (`abc123_my_change.py`) must be replaced.

## What Alembic Generates Correctly

Alembic autogenerate handles these reliably — review but generally trust:

- Adding new columns (nullable or with `server_default`)
- Creating new tables with columns, primary keys, and foreign keys
- Dropping columns marked as nullable with no dependents
- Adding and dropping simple indexes

---

## What Needs Manual Review

Alembic autogenerate **does not** detect these — you must write them by hand or verify the generated output carefully:

- **Partial indexes** (`CREATE INDEX ... WHERE condition`) — autogenerate emits a plain index; you must add the `postgresql_where` argument manually.
- **CHECK constraints** — not detected; add with `op.create_check_constraint(...)`.
- **Column type changes** — autogenerate may emit a no-op or an incorrect cast; verify the generated `alter_column` call and add `existing_type` + `postgresql_using` as needed.
- **`server_default` changes** — autogenerate does not diff server defaults; add `op.alter_column(..., server_default=...)` by hand.
- **Enum type changes** — adding values to a Postgres `ENUM` requires `ALTER TYPE ... ADD VALUE` which Alembic does not generate.

## Rollback

Roll back one revision:

```bash
cd api && uv run alembic downgrade -1
```

Or via the make wrapper (once a `migrate-down` target is added):

```bash
make migrate-down
```

Every migration must have a working `downgrade()` function. Never leave it as `pass` unless the operation is truly irreversible (document it with a comment in that case).

---

## Railway Migration Execution

On Railway, migrations run automatically as part of the `api` service start command:

```
uv run alembic upgrade head && uv run uvicorn app.main:app ...
```

This means:
- Migrations run before the API accepts traffic on every deploy.
- If a migration fails, the deploy fails and the previous version stays up.
- The `worker` service does not run migrations — it shares the same DB and starts after `api` is healthy.

---

## Rules

- **One migration per PR.** Never batch multiple unrelated schema changes into one file.
- **Never edit an applied migration.** Once a migration has been applied to any environment (staging, production, or another developer's local DB), it is frozen. Create a new migration to fix it.
- **Never squash migrations during MVP.** History is cheap and squashing is risky. Revisit in Phase 2.
- **Always commit migration + model change together.** A model without its migration (or vice versa) will break the next developer's `make migrate`.
